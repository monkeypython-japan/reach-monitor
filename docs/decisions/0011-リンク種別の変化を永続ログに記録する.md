# ADR-0011: リンク種別の変化を永続ログに記録する

- ステータス: 承認 (Accepted)
- 日付: 2026-07-16

## コンテキスト

subtitle-translation パイプラインが使う NAS（DXP4800PLUS）宛の SMB 接続で、再接続
（`smb_iod_start_reconnect`）が頻発する問題を調査中、ReachMonitor のメニューバー色
（[[0009-到達可能時のメニューバー丸色をリンク種別で出し分ける|ADR-0009]] により
赤=Ethernet相当／青=Wi-Fi）で「本来 Ethernet が優先されるはずなのに、一瞬 Wi-Fi に
なってまた戻る」というフラップを目視で確認した。

これは [[0008-nwpathmonitorで使用中のリンク種別を判定する|ADR-0008]] で `LinkMonitor`
（`NWPathMonitor`）が検知している経路変化そのものであり、SMB クライアント（特に SMB3
のマルチチャネル）は経路変化に反応して再接続することが知られているため、SMB 再接続
問題との相関が疑われている。

しかし ReachMonitor は表示のみで履歴を残さないため、過去に目視したフラップの発生時刻を
遡って確認できない。subtitle-translation 側で暫定的に `route -n get default` を1秒間隔で
ポーリングする外付けログ（`~/Library/Logs/iface_watch.log`）を仕込んだが、ReachMonitor は
既にイベント駆動でより正確な情報（`NWPathMonitor` の変化イベント + Wi-Fi 情報）を持って
いるため、アプリ本体にログ機能を持たせる方が望ましい。

## 検討した選択肢

- **採用案**: `LinkHistoryLogger` を新設し、`AppState` が `LinkMonitor.onUpdate` で
  `interfaceType` の変化（前回値との差分）を検知するたびに `~/Library/Logs/
  reachmonitor_history.log` へ1行追記する。同時点の `WiFiInfo`（SSID/BSSID/
  `hasWiFiInterface`）も併記し、「特定 AP へのローミングが原因か、単なる経路再評価か」
  を切り分けられるようにする。ログサイズが 5MB を超えたら `.log.1` へリネームして
  ローテートする、`FileHandle` 追記ベースのシンプルな実装とする。
- **却下案 A**: subtitle-translation 側の `route -n get default` ポーリング
  （`iface_watch.log`）をそのまま使い続ける — 却下理由: 1秒ポーリングのため
  瞬間的なフラップを取りこぼす可能性があり、ReachMonitor が `NWPathMonitor` の
  イベントで既に正確に検知できている情報を二重に外付けで推測する形になる。
- **却下案 B**: `LinkMonitor` 自身にログ機能を持たせ、変化判定も `LinkMonitor` 内で
  完結させる — 却下理由: 到達性のエッジ検出・通知発火が既に `AppState.apply()` に
  集約されている（CLAUDE.md: 「AppState はエッジを判断する唯一の場所」）のと同じ
  パターンに揃え、`LinkMonitor` は素の値を報告するだけの薄いモニターのまま保つ方が
  既存の設計と一貫する。

## 決定

`Sources/ReachMonitor/LinkHistoryLogger.swift` を新設。`AppState` の
`linkMonitor.onUpdate` コールバック内で、新しい `LinkInfo.interfaceType` が直前の値と
異なる場合にのみ `linkHistory.logLinkChange(from:to:wifi:)` を呼び出す（変化がない
呼び出しでは記録しない）。ログの追記・ローテーション処理は同期的な `FileHandle` 操作
とし、失敗（権限エラー等）は無視してアプリの動作に影響させない。

## 結果

### 利点

- メニューバーで目視した「一瞬のフラップ」を、正確なタイムスタンプ（ミリ秒精度）と
  Wi-Fi 情報付きで後から確認できる。
- subtitle-translation 側の `smb_watch.log` と時刻を突き合わせることで、経路フラップと
  SMB 再接続の相関を検証できる。
- 到達性の通知（`NotificationManager`）と同じ「`AppState` がエッジを検知して副作用を
  呼び出す」パターンに揃えたため、コードの見通しが既存実装と一貫している。

### 欠点・トレードオフ

- ログファイルへの書き込みはメインアクター上で同期的に行われる（`FileHandle` の
  open/seek/write）。個人利用ツールでイベント頻度も低いため許容しているが、頻繁に
  フラップする環境ではメインスレッドを短時間ブロックしうる。
- ローテーションは世代数1（`.log.1` のみ）。長期のログ保全は想定していない。

## 備考

  （subtitle-translation 側の `~/Library/Logs/smb_watch.log` と同じディレクトリ）
