# ADR-0002: 到達判定に TCP 接続テストを採用する

- ステータス: 承認 (Accepted)
- 日付: 2026-07-01

## コンテキスト

スタバ・マクドナルド等の共有 Wi-Fi では、AP に接続（アソシエーション）できても captive
portal 未ログインや上流障害により目的アドレスへの実到達性がないことがある。「目的アドレスに
リーチできているか」をどう判定するかを決める必要があった。

## 検討した選択肢

- **採用案**: `NWConnection` による TCP 接続テスト（`1.1.1.1:443` / `8.8.8.8:53` /
  `apple.com:443` へ実際に接続を試みる）。
- **却下案 A**: ICMP ping — 却下理由: 到達可否と RTT は分かるが、captive portal
  環境では ICMP がフィルタされていることが多く、またポート単位の到達性
  （実際に使うプロトコルが通るか）を検証できない。
- **却下案 B**: HTTPS リクエストで応答内容まで検証する（例: `captive.apple.com` への
  GET でリダイレクト有無を見る） — 却下理由: captive portal 検出の精度は上がるが、
  実装コストが TCP 接続テストより高く、初期実装としては TCP ハンドシェイクの成否で
  十分と判断した。将来 captive portal 誤検知（TCP は通るが実際は portal 画面、
  詳細は [[swift-run-では位置情報-通知が機能しない|注意点]] 参照）が問題になれば
  ADR を起こして再検討する。
- **監視対象**: 設定 UI は作らずコード内の定数（`Targets.swift` の `DefaultTargets.all`）で
  固定管理する — 個人利用でありデフォルト値の変更頻度が低いため。

## 決定

`ReachabilityMonitor` が `MonitorConfig.checkInterval`（既定 10 秒）ごとに、各 `Target` へ
`NWConnection` で TCP 接続を試み、いずれか1つでも `.ready` になれば `.reachable`、
全滅すれば `.unreachable` とする。

## 結果

### 利点

- 実装がシンプルで、captive portal 環境でもポート単位の実到達性を検証できる。
- 監視先を `Targets.swift` の配列に足すだけで拡張できる。

### 欠点・トレードオフ

- TCP ハンドシェイクだけ通り実際には captive portal のログイン画面という状態を
  「到達」と誤判定する可能性がある（既知の制約として README/CLAUDE.md に明記）。

## 備考

- 関連実装: `Sources/ReachMonitor/ReachabilityMonitor.swift`, `Sources/ReachMonitor/Targets.swift`
- 2026-07-22: 却下案 B で予告していた通り、captive portal による TCP ハンドシェイクの誤検知が
  実際に問題になったため、[[0013-https対象はtcp接続ではなくtls検証まで行う|ADR-0013]] で
  HTTPS 用ターゲットのみ TLS 証明書検証まで行うよう変更した（TCP 接続テストという方針自体は
  変わらず、HTTPS 対象の「到達」判定基準が強化された）。
