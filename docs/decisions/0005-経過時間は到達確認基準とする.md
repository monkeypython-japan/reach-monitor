# ADR-0005: メニューバーの経過時間表示は「到達確認」基準とする

- ステータス: 承認 (Accepted) — ただし適用範囲がポップオーバーに縮小
  （[[0012-メニューバーの経過時間表示をリンク接続かつ到達確認基準にする|ADR-0012]] 参照）
- 日付: 2026-07-07

## コンテキスト

当初の要望は「現在の AP に接続してからの経過時間」を表示することだった。実装当初は
`WiFiMonitor` が BSSID の変化を検知し、その時刻を `connectedSince` として経過時間の起点に
していた。

その後、経過時間の意味をより実用的なものに変更したいという要望が出た：
「経過時間は最初のリーチ確認からスタートし、リーチが切断された時点でタイマーを止め、
次にリーチが確認されたら 0:00 から再スタートする」。これは AP に繋がっているだけでは
分からない「実際にどれくらいの間、目的アドレスに到達できているか / できていないか」を
示す値であり、このアプリの本来の目的（captive portal 等での実到達性監視）により合致する。

## 検討した選択肢

- **採用案**: `AppState` が到達状態の変化を追跡し、`reachTimerStart`（到達中はここから
  カウント）と `reachTimerFrozenElapsed`（到達を失った瞬間に固定される値）で経過時間を
  管理する。`checkNow()`（手動再チェック）が一時的に `status = .checking` にする際に
  誤ってタイマーがリセットされないよう、実際に確定した到達状態のみを追う
  `lastConfirmedStatus` を `status` とは別に保持する。
- **却下案（旧実装）**: `WiFiMonitor` の BSSID 変化時刻（AP 接続時刻）を経過時間の起点にする
  — 却下理由: AP に接続できているだけでは目的アドレスへの実到達性は分からず、本来
  知りたい情報（到達できているか）とずれる。ユーザからの明示的な変更依頼により置き換えた。

## 決定

経過時間は次の規則で管理する:
- 最初に到達 (`.reachable`) が確認された時点、または `.unreachable` からの復旧時点で
  `reachTimerStart = Date()`（0:00 から再スタート）。
- 到達から非到達へ遷移した瞬間に、その時点の経過秒数を `reachTimerFrozenElapsed` に固定し
  `reachTimerStart` をクリアする（タイマー停止）。
- 同一状態が続く間は値を変えない（到達中はライブでカウントアップ、非到達中は固定値のまま）。
- 手動「今すぐ再チェック」による一時的な `.checking` 状態はこのロジックに影響しない。

`AppState.reachElapsedSeconds` を単一の情報源とし、メニューバー (`h:mm`) とポップオーバー
(`hh:mm:ss`) はこの値をフォーマットのみ変えて表示する。

## 結果

### 利点

- 「今どれだけの時間、目的アドレスに到達できているか／できていないか」が一目で分かる、
  アプリの目的に合致した指標になった。
- 手動再チェックでタイマーが暴発的にリセットされない。

### 欠点・トレードオフ

- AP 接続時刻という情報は表示されなくなった（`WiFiMonitor.connectedSince` は削除済み）。
  今後 AP 切替そのものの情報が必要になった場合は再実装が必要。

  ポップオーバーの経過時間表示は本 ADR の到達確認基準のまま変更していない
  （メニューバーと同じ値を出すと冗長なため、意図的に別の指標のまま残した）。
