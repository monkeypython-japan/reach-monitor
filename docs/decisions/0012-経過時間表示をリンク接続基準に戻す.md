# ADR-0012: メニューバーの経過時間表示だけをリンク接続基準に戻す

- ステータス: 承認 (Accepted)
- 日付: 2026-07-17

## コンテキスト

[[0011-リンク種別の変化を永続ログに記録する|ADR-0011]] の文脈と同じ、subtitle-translation
パイプラインの NAS 宛 SMB 再接続問題の調査の一環。メニューバーの経過時間は
[[0005-経過時間は到達確認基準とする|ADR-0005]] により「到達確認からの経過時間」（TCP
到達性ベース）だったが、これでは Wi-Fi⇄Ethernet の瞬間的なリンクフラップがあっても
到達性（`.reachable`）自体は途切れないまま推移してしまうことがあり、「今どれだけの間、
現在のリンクに安定して繋がっているか」が経過時間から読み取れなかった。

ユーザーはメニューバーの経過時間表示を、SMB 再接続の相関調査に使える「現在のリンクに
接続してからの経過時間」に変更したいという明示的な要望を出した。リンクが切れたら停止・
新しいリンクが繋がったら 0:00 から再スタートし、そのイベントを ADR-0011 のログにも記録する。

当初、ポップオーバーの経過時間表示（`hh:mm:ss`）も同じ基準に揃える実装をしたが、
「ポップオーバーは到達確認基準の継続時間のままでよく、メニューバーと重複した表示は
冗長になる」というフィードバックを受け、**メニューバーのみ**をリンク接続基準に切り替える
方針に変更した。

## 検討した選択肢

- **採用案**: メニューバーの経過時間（`h:mm`）だけをリンク接続基準に切り替え、ポップオーバーの
  経過時間（`hh:mm:ss`、「到達確認からの経過時間」）は [[0005-経過時間は到達確認基準とする|
  ADR-0005]] のまま維持する。`AppState` に2つの独立したタイマー状態を持たせる：
  - `linkTimerStart`/`linkTimerFrozenElapsed`（リンク接続基準、`linkMonitor.onUpdate` の
    `interfaceType` エッジで更新） → メニューバーの `menuBarElapsedText(at:)` が使う。
  - `reachTimerStart`/`reachTimerFrozenElapsed`（到達確認基準、`apply(results:status:)` の
    `ReachStatus` エッジで更新） → ポップオーバーの `ElapsedTimeRow` が使う。
  リンクのリセット/凍結イベントは `LinkHistoryLogger` の同じログ行に `timer=reset` /
  `timer=freeze(upSeconds=N)` として記録する。
- **却下案 A**: 両方の表示をリンク接続基準に統一する — 却下理由: ユーザーから明示的に
  「ポップオーバーは到達確認基準のままでよい、メニューバーと重複すると冗長」との
  フィードバックがあった。到達性そのもの（「今どれだけ実際に目的アドレスへ到達できて
  いるか」）を示す指標が失われてしまう。
- **却下案 B**: 到達性ベースの経過時間表示は残しつつ、リンク接続時間を別の新しい行として
  ポップオーバーに追加表示する — 却下理由: メニューバーに既にリンク接続基準の経過時間が
  表示されるため、ポップオーバーにも同じ情報を出すと冗長（ユーザーからのフィードバックの
  趣旨そのもの）。

## 決定

`AppState` は `linkTimerStart`/`linkTimerFrozenElapsed` と `reachTimerStart`/
`reachTimerFrozenElapsed` を両方持つ。前者は `linkMonitor.onUpdate` 内で
`interfaceType` の変化を検出するたびに `updateLinkTimer(newType:)` で更新し、
`linkElapsedSeconds(at:)` → `menuBarElapsedText(at:)` を通じてメニューバーのみに反映する。
後者は既存通り `apply(results:status:)` 内の到達性エッジで更新し、`reachElapsedSeconds(at:)`
を通じてポップオーバーの `ElapsedTimeRow` にのみ反映する。両者は完全に独立しており、
一方が他方を駆動することはない。

```swift
private func updateLinkTimer(newType: NWInterface.InterfaceType?) -> LinkHistoryLogger.TimerEvent {
    if newType != nil {
        linkTimerStart = Date()          // 新規接続・種別切替: 0:00 から再スタート
        linkTimerFrozenElapsed = nil
        return .reset
    } else {
        if let start = linkTimerStart {
            linkTimerFrozenElapsed = Date().timeIntervalSince(start)
        }
        linkTimerStart = nil              // リンク断: 凍結
        return .freeze(elapsedSeconds: max(0, Int(linkTimerFrozenElapsed ?? 0)))
    }
}
```

## 結果

### 利点

- メニューバーの経過時間が「今のリンクにどれだけ安定して繋がっているか」を正確に反映し、
  SMB 再接続との相関調査にそのまま使える指標になった。
- ポップオーバーは従来通り「今どれだけ実際に目的アドレスへ到達できているか」を示し続け、
  アプリ本来の目的（到達性監視）の指標も失われない。
- 2つの表示が異なる情報を示すため、メニューバーとポップオーバーを両方確認することで
  「リンクは繋がっているが到達性はまだ確認できていない」「到達性はあるがリンクは
  最近フラップした」といった状況の切り分けができる。
- リセット/凍結イベントが [[0011-リンク種別の変化を永続ログに記録する|ADR-0011]] のログに
  併記されるため、「いつメニューバーのタイマーが 0:00 に戻ったか」を後から突き合わせられる。

### 欠点・トレードオフ

- `AppState` が経過時間用のタイマー状態を2組（`linkTimer*`/`reachTimer*`）持つことになり、
  「メニューバーとポップオーバーで見えている経過時間の意味が違う」という非自明な仕様が
  増えた（CLAUDE.md の Elapsed-time semantics セクションに明記して緩和している）。
- Wi-Fi と Ethernet の間で頻繁にフラップする環境では、メニューバーの経過時間が 0:00 付近を
  往復し続ける（これ自体がフラップの検出に役立つ指標として意図されている）。
