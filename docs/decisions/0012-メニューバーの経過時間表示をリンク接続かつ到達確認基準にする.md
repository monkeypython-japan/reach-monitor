# ADR-0012: メニューバーの経過時間表示を「リンク接続 かつ 到達確認」基準にする

- ステータス: 承認 (Accepted)
- 日付: 2026-07-17

## コンテキスト

[[0011-リンク種別の変化を永続ログに記録する|ADR-0011]] の文脈と同じ、subtitle-translation
パイプラインの NAS 宛 SMB 再接続問題の調査の一環。メニューバーの経過時間は
[[0005-経過時間は到達確認基準とする|ADR-0005]] により「到達確認からの経過時間」（TCP
到達性ベース）だったが、これでは Wi-Fi⇄Ethernet の瞬間的なリンクフラップがあっても
到達性（`.reachable`）自体は途切れないまま推移してしまうことがあり、「今どれだけの間、
現在のリンクに安定して繋がっているか」が経過時間から読み取れなかった。

ユーザーはメニューバーの経過時間表示を、SMB 再接続の相関調査に使える指標に変更したいという
明示的な要望を出した。検討は次の2段階で進んだ:

1. まずリンク接続基準（`LinkMonitor` の `interfaceType` 変化）だけに切り替える実装をした。
2. その後「到達確認ができなかった場合もメニューバーの経過時間はリセットされるべきでは」との
   確認があり、「到達確認が失敗した時点で停止（凍結）し、到達が復旧したら 0:00 から
   再スタート」という到達性エッジの挙動を、リンクエッジの挙動と併せて持つことになった。

また、当初ポップオーバーの経過時間表示（`hh:mm:ss`）も同じ基準に揃える実装をしたが、
「ポップオーバーは到達確認基準の継続時間のままでよく、メニューバーと重複した表示は
冗長になる」というフィードバックを受け、**メニューバーのみ**を変更する方針にした。

## 検討した選択肢

- **採用案**: メニューバーの経過時間（`h:mm`）は「リンク接続 かつ 到達確認」の両方が
  成立している状態（＝"fully healthy"）が続いている時間とする。リンクが切れる・
  到達確認に失敗する、どちらか一方でも起きた時点で凍結し、新しいリンクへの接続・
  到達性の復旧のどちらか一方でも起きた時点で 0:00 から再スタートする。ポップオーバーの
  経過時間（`hh:mm:ss`、「到達確認からの経過時間」）は
  [[0005-経過時間は到達確認基準とする|ADR-0005]] のまま、到達性のみを基準に維持する。
  `AppState` は2つの独立したタイマー状態を持つ：
  - `menuBarTimerStart`/`menuBarTimerFrozenElapsed`（fully-healthy 基準。
    `linkMonitor.onUpdate` の `interfaceType` エッジと `apply(results:status:)` の
    `ReachStatus` エッジの**両方**から `resetMenuBarTimer()`/`freezeMenuBarTimer()` 経由で
    更新される） → メニューバーの `menuBarElapsedText(at:)` が使う。
  - `reachTimerStart`/`reachTimerFrozenElapsed`（到達確認基準のみ、`apply(results:status:)`
    の `ReachStatus` エッジのみで更新） → ポップオーバーの `ElapsedTimeRow` が使う。
  `resetMenuBarTimer()`/`freezeMenuBarTimer()` は冪等（既に同じ状態への呼び出しは安全に
  no-op）にし、2つのエッジ源からの呼び出し順序を気にせず合成できるようにする。
- **却下案 A**: リンク接続基準のみ（到達性エッジは無視）— 却下理由: ユーザーから
  「到達確認ができなかった場合もリセットされるべき」との明示的な追加要望があった。
- **却下案 B**: 両方の表示を fully-healthy 基準に統一する — 却下理由: ポップオーバーは
  到達確認基準のままでよいという既存のフィードバックと矛盾する。到達性そのもの
  （「今どれだけ実際に目的アドレスへ到達できているか」）を示す指標が失われてしまう。
- **却下案 C**: リンクエッジと到達性エッジをそれぞれ別のタイマー変数で持ち、メニューバーは
  「両方が非凍結の時だけ加算する」独自の合成ロジックにする — 却下理由: 単純な
  reset/freeze の呼び出し合成（採用案）で同じ結果が得られ、状態変数も1組で済む。
  「個人利用ツールなので凝った設計は不要」という既存方針にも合う。

## 決定

`AppState.menuBarTimerStart`/`menuBarTimerFrozenElapsed` を単一の状態として持ち、
`resetMenuBarTimer()`/`freezeMenuBarTimer()` という2つの小さなヘルパーを、リンクエッジ
（`updateLinkTimer(newType:)`）と到達性エッジ（`apply(results:status:)` 内）の両方から呼ぶ。

```swift
private func resetMenuBarTimer() {
    menuBarTimerStart = Date()
    menuBarTimerFrozenElapsed = nil
}

private func freezeMenuBarTimer() {
    if let start = menuBarTimerStart {
        menuBarTimerFrozenElapsed = Date().timeIntervalSince(start)
    }
    menuBarTimerStart = nil
}
```

`updateLinkTimer(newType:)`（`interfaceType` が実際に変化した時のみ呼ばれる）は新規接続で
`resetMenuBarTimer()`、リンク断で `freezeMenuBarTimer()` を呼ぶ。`apply(results:status:)`
は到達性の復旧で `resetMenuBarTimer()`、到達性の喪失で `freezeMenuBarTimer()` を呼ぶ
（ポップオーバー用の `reachTimerStart`/`reachTimerFrozenElapsed` の更新に追加する形）。
凍結済みの状態への `freezeMenuBarTimer()` はそのまま値を保持し、リセットは最後に呼ばれた
方が新しい起点になる — 2つのエッジ源の呼び出し順序を気にする必要がない。

`LinkHistoryLogger` へのログ記録（`timer=reset`/`timer=freeze(upSeconds=N)`）は
リンクエッジ側のみで行う。到達性エッジによるリセット/凍結はログに記録しない
（ADR-0011 のログはリンク/インターフェース変化に絞ったものであり、TCP 到達性の失敗は
ドット色や通知で既に可視化されているため）。

## 結果

### 利点

- メニューバーの経過時間が「リンクも到達性も安定している時間」を正確に反映し、
  SMB 再接続との相関調査にそのまま使える指標になった。
- ポップオーバーは従来通り「今どれだけ実際に目的アドレスへ到達できているか」を示し続け、
  アプリ本来の目的（到達性監視）の指標も失われない。
- 2つの表示が異なる情報を示すため、メニューバーとポップオーバーを両方確認することで
  状況の切り分けができる（例: ポップオーバーは長時間到達可能を示しているのにメニューバーが
  頻繁に 0:00 に戻る＝リンクだけがフラップしている、等）。
- `resetMenuBarTimer()`/`freezeMenuBarTimer()` を冪等にしたことで、2つの独立したエッジ源から
  呼ばれても状態が壊れない。

### 欠点・トレードオフ

- `AppState` が経過時間用のタイマー状態を2組（`menuBarTimer*`/`reachTimer*`）持つことになり、
  「メニューバーとポップオーバーで見えている経過時間の意味が違う」という非自明な仕様が
  増えた（CLAUDE.md の Elapsed-time semantics セクションに明記して緩和している）。
- メニューバーのタイマーが「リンクと到達性のどちらのエッジで最後にリセット/凍結されたか」を
  区別しない（合成された単一の値のため）。何が原因だったかは
  `~/Library/Logs/reachmonitor_history.log`（リンクエッジのみ）や通知履歴と突き合わせて
  判断する必要がある。
- Wi-Fi と Ethernet の間で頻繁にフラップする環境、または到達確認が不安定な環境では、
  メニューバーの経過時間が 0:00 付近を往復し続ける（これ自体がフラップ・不安定性の検出に
  役立つ指標として意図されている）。
